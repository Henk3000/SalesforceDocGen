Architecture Overview

The Salesforce DocGen Platform is a native Salesforce solution designed for high-performance document generation. It follows a modular architecture that separates data retrieval, document processing, and UI orchestration.

Bulk Processing Pipeline

To handle hundreds of documents without hitting governor limits, the system uses a high-performance asynchronous pipeline.

How it Works

  1. Job Initiation: A user selects a template and defines a filter (or uses a saved query).
  2. Job Creation: A DocGen_Job__c record is created to track progress.
  3. Stateful Batching: The DocGenBatch runs with a deliberate Batch Size of 10. This ensures we don't exceed the 10MB heap limit or CPU timeouts while performing complex ZIP compression.
  4. Data Safety: The DocGenDataRetriever uses manual field mapping instead of JSON serialization to further reduce memory spikes.

Background PDF Engine

Generating PDFs in the background (Batch/Flow) without external dependencies requires a specialized "Loopback" architecture to capture Salesforce's native file renditions.

The Dynamic Loopback Workflow

Because PDF conversion is not natively exposed to Apex, we leverage the internal Connect API via a secure authenticated callout back to the same org.

Zero-Config Deployment: The package dynamically generates a loopback address using Url.getOrgDomainUrl().toExternalForm(). This eliminates the need for any hardcoded Named Credentials or Remote Site Settings in subscriber orgs.

Asynchronous Retry Mechanism for "202 Accepted"

The Salesforce Connect API is asynchronous. When requesting a PDF, the API may return a 202 Accepted status if the rendition is still processing.

To handle this latency, the DocGenRenditionQueueable class incorporates a recursive retry pattern. If a 202 status is encountered, the job catches a custom RenditionException and resubmits itself to the Queueable stream (up to 3 times), ensuring the PDF is captured when ready.

"Smart" Table Row Expansion

One of the most complex features is the table logic. If a loop placeholder (e.g., {#Items}) is detected inside a Word Table Row (<w:tr>), the engine:

  1. Identifies the XML boundaries of that specific row.
  2. Strips the loop tags from the row XML.
  3. Duplicates the row XML for every record in the collection.
  4. Merges the results back into the document XML.

How to Improve

This is a first iteration. Here are areas where you can contribute or extend the project:

  • Complex Conditionals: Add support for nested {#if} logic in the XML parser.
  • PDF Rendering: Improve the server-side PDF conversion (currently handled via Visualforce/JS).
  • Multi-Object Joins: Enhance the Query Builder to support complex multi-level lookups and child relationships beyond 2 levels.
  • Custom Hook Interface: Create a standard interface for "Preprocessing" data before it hits the engine.

Roadmap: Server-Side PDF Generation

To move beyond browser-side "tricks" and enable true background PDF generation (Bulk/Flow), consider the following production-grade strategies.

Option A: The Cloud API Strategy (Recommended)

Integrate with a specialized document conversion API via Apex callouts.

  • CloudConvert / Cloudmersive: Send the DOCX blob, get the PDF blob back.
  • Docmosis: Move the entire merge + PDF conversion logic to their cloud. This is how Salesforce OmniStudio/Vlocity actually works under the hood.
  • Pros: Zero server maintenance, highest formatting fidelity.

Option B: The "Serverless" Worker (Advanced)

Build a custom microservice in AWS Lambda or Heroku running Gotenberg or LibreOffice Headless.

  • Mechanism: Apex calls your Lambda; Lambda runs a real Office engine and returns the PDF.
  • Pros: Lowest per-document cost for very high volumes.

Option C: The Google Drive "Conversion" Hack

Upload the DOCX to a Service Account's Google Drive, then export it as a PDF.

  • Pros: Uses Google's world-class rendering engine.
  • Cons: High complexity (Auth/Multi-step callouts).
